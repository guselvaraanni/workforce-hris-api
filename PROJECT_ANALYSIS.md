## 1. Project Overview

### 1.1 What this project does

`workforce-hris-api` is a **Django + Django REST Framework** Human Resources
Information System. It provides:

- Authentication (JWT for the API, session login for the server-side UI).
- Role-aware management of **Employees**, **Departments**, **Leave Requests**,
  **Audit Logs**, and **Bulk CSV Uploads**.
- Two parallel front-ends:
  - **Server-side Django templates** under `/dashboard/`, `/employees/`, etc.
  - A **partial SPA shell** (`templates/index.html`) used only for the login
    landing page, plus a legacy `static/js/app.js` (unwired — see §10).
- Background CSV ingestion using **`threading.Thread`** (no Celery/Redis).
- A `WorkforceAnalyticsView` that aggregates DB-level stats.

### 1.2 Main business purpose

Internal HR tooling: HR admins onboard / off-board employees, managers approve
their reports' leave, employees submit leave and edit their own profile.

### 1.3 Key workflows

1. **Login** → `/` renders `index.html` → POST `/login/` (session) → redirect to
   `/dashboard/`.
2. **HR adds an employee** → `/employees/add/` form → `POST /api/v1/employees/`
   (multipart) → creates `CustomUser` + `EmployeeProfile` + `AuditLog`.
3. **Employee submits leave** → quick form on dashboard or `/leaves/` →
   `POST /api/v1/leave-requests/`.
4. **Manager / HR approves leave** → `/leaves/` →
   `POST /api/v1/leave-requests/{id}/approve_leave/`.
5. **HR bulk-imports employees** → `/uploads/` drag-drop CSV →
   `POST /api/v1/bulk-uploads/` → spawns background thread → polls status.
6. **HR reviews audit trail** → `/audit/` (template) and
   `GET /api/v1/audit-logs/` (API).

### 1.4 Architecture summary

```
┌─────────────────────────────┐
│  Browser                    │
│  ├─ index.html (login SPA)  │
│  └─ Django templates        │── (server-rendered base.html shell + JS fetch())
└──────────────┬──────────────┘
               │ HTTP/HTTPS
┌──────────────▼──────────────┐
│  Django 4.2 (config/)       │
│  ├─ Template views (CBV)    │── reads DB, renders HTML
│  └─ DRF ViewSets + APIView  │── JSON API at /api/v1/*
│        ├─ JWT SimpleJWT     │
│        ├─ Session auth      │
│        ├─ DjangoFilterBackend
│        └─ Pagination 20     │
└──────────────┬──────────────┘
               │ ORM (mysqlclient)
┌──────────────▼──────────────┐
│  MySQL 8.0                  │
│  └─ employee_management_db  │
└─────────────────────────────┘

Background thread (started via transaction.on_commit) → process_csv_file()
```

The system has **no Celery, no Redis, no message queue**. Background work is a
`threading.Thread` daemon=False — this works in dev but is fragile in
production (no retry, no visibility, killed by Gunicorn worker recycle).

- **`templates/partials/` is empty** — referenced nowhere.

  `.gitignore` only ignores `*.log` so it _is_ ignored in git, but is a
  reproducibility hazard).

## 4. Backend API Documentation

All endpoints live under `/api/v1/`. Auth class default = JWT + Session;
permission default = `IsAuthenticated`.

### 4.1 Authentication APIs

#### `POST /api/v1/auth/login/`
- **View**: `CustomTokenObtainPairView` (`views.py:80`)
- **Purpose**: JWT login. Returns `access`, `refresh`, plus custom claims.
- **Payload**: `{ "email": "...", "password": "..." }` (uses `USERNAME_FIELD='email'`)
- **Response 200**: `{ access, refresh, user_id, email, full_name, is_hr_admin, is_manager }`
- **Auth**: `AllowAny`
- **Frontend integration**: **NONE**. The login page (`index.html`) instead
  posts to `/login/` (session-based). The JWT endpoint is **never called by
  any UI in this repo**.
- **Issues**: Two near-identical serializers exist (`CustomTokenObtainPairSerializer`
  in `views.py` AND `serializers.py`) — duplicate code.

#### `POST /api/v1/auth/refresh/`
- **View**: `TokenRefreshView` (SimpleJWT default)
- **Frontend integration**: Used only by `static/js/app.js` which itself is
  unwired. **Dead in practice.**

#### `POST /api/v1/auth/register/`
- **View**: `UserRegistrationView` (`views.py:113`)
- **Auth**: `AllowAny` — **SECURITY ISSUE**: anyone on the internet can create
  a `CustomUser`. Together with the model's `post_save` signal (see Bug #1),
  this will *also* automatically create an `EmployeeProfile` with `salary=0`.
- **Frontend integration**: **NONE.** No registration UI is built.

#### `GET /api/v1/auth/users/`
- **View**: `UserListView` (`views.py:119`)
- **Permission**: `IsHRAdmin`
- **Issues**: Declares `search_fields` and `ordering_fields` but uses
  `ListAPIView`, not a ViewSet, so `DEFAULT_FILTER_BACKENDS` apply — fine. Not
  consumed by any UI.

#### `POST /login/`
- **View**: `SessionLoginView` (`views.py:85`)
- **Purpose**: Session-cookie login used by `index.html`.
- **Issues**:
  - Returns just `{ detail }` — no user info. UI hard-codes the redirect.
  - Doesn't return user role flags, so the front end cannot personalise the
    next request before its first server-rendered page.
  - `@method_decorator(csrf_protect)` is fine, but the page also needs the
    user to have a CSRF cookie set before the first POST — works because
    `TemplateView` for `/` renders the form via Django and that issues a
    cookie.

#### `GET /logout/`
- **View**: `LogoutView` (Django built-in) → next_page `/`.
- **Frontend integration**: `base.html` calls it via JS (`logoutAction`).

### 4.2 Department APIs (`DepartmentViewSet`)

Router base: `/api/v1/departments/`

| Method | Path | Permission | UI |
|---|---|---|---|
| GET   | `/`           | Auth          | `/departments/` template |
| POST  | `/`           | `IsHRAdmin`   | **NOT WIRED**. There is no UI to create a department. (Only auto-created during CSV upload via `Department.objects.get_or_create`.) |
| GET   | `/{id}/`      | Auth          | Not used (no detail page) |
| PATCH | `/{id}/`      | `IsHRAdmin`   | **NOT WIRED**. No UI to edit. |
| DELETE| `/{id}/`      | `IsHRAdmin`   | **NOT WIRED**. No UI to delete. |

Queryset is properly optimised:
- `select_related('head')`
- `annotate(active_employee_count=Count('employees', filter=Q(employees__employment_status='active')))`

Audit logging via `log_action()` in `perform_create` and `perform_update`.
`perform_destroy` is **not overridden** — so deleting a department writes
**no audit log** (a real audit-trail hole).

### 4.3 Employee APIs (`EmployeeProfileViewSet`)

Router base: `/api/v1/employees/`

| Method | Path | Permission | UI |
|---|---|---|---|
| GET | `/` | Auth (queryset is role-filtered) | `/employees/` template (server-rendered, not the API) |
| POST | `/` | `IsHRAdmin` | `/employees/add/` page (calls the API via fetch) |
| GET | `/{id}/` | Auth + queryset filter | Used implicitly by template detail (template loads from DB, not the API) |
| PATCH | `/{id}/` | `IsOwnEmployeeProfileOrAdmin` | Edit page submits a Django form to `EmployeeUpdateView`, **NOT** the API |
| DELETE | `/{id}/` | `CanDeleteEmployee` (HR only) | **NOT WIRED** — no delete button in any template |
| GET | `/by_department/?department_id=…` | `IsHRAdmin` | **NOT WIRED** |
| GET | `/my_team/` | `IsManager` | **NOT WIRED** (dashboard shows team via direct ORM, not the API) |

**Critical defects in this resource (see §10):**
1. `POST /api/v1/employees/` always raises **IntegrityError** because of the
   `CustomUser.post_save` signal duplicating the profile insert.
2. Search `search_fields` set on the ViewSet are good, **but** `filterset_fields`
   is set as a list — overrides any `EmployeeFilter` from `filters.py`, so
   `filters.py::EmployeeFilter` is dead code.
3. Custom action `by_department` correctly paginates, but the action
   `my_team` is decorated with `permission_classes=[IsAuthenticated, IsManager]`
   — yet HR admins (who are not flagged as managers) cannot use it. Fine
   semantically but inconsistent with the queryset's `if user.is_hr_admin:
   return qs` rule.
4. `perform_update` re-fetches the old object then calls `serializer.save()`.
   Because `pre_save` + `post_save` signals also fire, this creates **two
   audit log rows per update** (one explicit via `log_action`, one via
   `signals.py::log_employee_profile_changes`).

### 4.4 Leave Request APIs (`LeaveRequestViewSet`)

Router base: `/api/v1/leave-requests/`

| Method | Path | Permission | UI |
|---|---|---|---|
| GET | `/` | Auth (queryset role-filtered) | `/leaves/` template (server-side) |
| POST | `/` | Auth | `/leaves/` form, dashboard quick form |
| GET | `/{id}/` | Auth | not used |
| PATCH | `/{id}/` | Auth (default) | **Exposed but no validation enforces** that fields like `status` can only be changed via the action endpoints — a regular employee can `PATCH` their own leave to `approved` because the serializer's `read_only_fields` is `['id','employee','created_at','updated_at','approved_by','approved_by_name','approval_date']` but **`status` is writable**. **SECURITY BUG.** |
| DELETE | `/{id}/` | Auth | **NOT WIRED** in UI. Also: any authenticated employee may delete any leave that their queryset returns. |
| POST | `/{id}/approve_leave/` | `CanApproveLeaves` | `/leaves/` "Approve" button (JS prompt) |
| POST | `/{id}/reject_leave/` | `CanApproveLeaves` | `/leaves/` "Reject" button |
| GET | `/my_requests/` | Auth | **NOT WIRED** (template uses ORM directly) |
| GET | `/pending_approvals/` | `IsHRAdminOrManager` | **NOT WIRED** in UI |

**Defects:**
- **Overlapping leave validation is a TODO** (`serializers.py:208`) but
  `tests/test_complete.py::test_overlapping_leave_rejected` *asserts* it
  rejects. The test will **fail**.
- `LeaveRequestSerializer.validate` performs *side effects* (creates an
  `EmployeeProfile` via `get_or_create`). Validators must be pure.
- `filterset_fields` is hard-coded again, so `filters.py::LeaveFilter` is dead.
- `CanApproveLeaves.has_object_permission` requires
  `hasattr(request.user, 'subordinates')` — but `subordinates` is a *reverse
  manager*ForeignKey related_name, which **always exists** on every user. So
  this check is a no-op. The intent — "user must have at least one direct
  report" — is not enforced.
- `select_for_update()` is used correctly inside `transaction.atomic` for the
  approve/reject race. ✅
- The `signals.py::log_leave_request_changes` post_save signal will
  *additionally* write an `AuditLog` whenever status changes, so each
  approve/reject creates **two** audit log rows.

### 4.6 Bulk Upload APIs (`BulkUploadViewSet`)

Router base: `/api/v1/bulk-uploads/`

| Method | Path | Permission | UI |
|---|---|---|---|
| GET | `/` | `IsHRAdmin` | `/uploads/` (template uses ORM directly) |
| POST | `/` | `IsHRAdmin` | `/uploads/` drag-drop |
| GET | `/{id}/` | `IsHRAdmin` | Not used |
| GET | `/{id}/status/` | `IsHRAdmin` | **NOT WIRED** (page reloads instead) |
| POST | `/{id}/retry/` | `IsHRAdmin` | `/uploads/` "Retry" button |
| DELETE | `/{id}/` | `IsHRAdmin` | **NOT WIRED** |

Strengths:
- `transaction.on_commit(...)` properly defers thread start until commit.
- `process_csv_file()` performs a two-phase validate/insert with bulk_create.
- Duplicate handling: in-DB and in-CSV.
- Audit log entries bulk-created for ingested rows.

Weaknesses:
- The thread is `daemon=False` so it survives Gunicorn worker recycle —
  but it **does not survive process exit gracefully** (no `SIGTERM` handling).
- No retry on transient DB errors.
- `csv_file.seek(0); csv_file.read().decode('utf-8')` loads the entire CSV
  into memory — defeats the "streaming" claim in the docstring.
- `CustomUser.objects.bulk_create(...)` does **not** trigger the post_save
  signal — so for CSV-imported users, no `EmployeeProfile` is auto-created
  by the signal, which is **good** for this flow but inconsistent with API
  creation.
- `make_random_password()` is removed in Django 5 — works in 4.2 only.

## 5. Frontend Analysis

### 5.1 Main pages (template-driven)

| URL | Template | View class |
|---|---|---|
| `/` | `index.html` | `TemplateView` (login page) |
| `/dashboard/` | `dashboard.html` | `DashboardView` |
| `/employees/` | `employees.html` | `EmployeeListView` |
| `/employees/<uuid:pk>/` | `employee_detail.html` | `EmployeeDetailView` |
| `/employees/add/` | `employee_form.html` | `EmployeeCreateView` |
| `/employees/<uuid:pk>/edit/` | `employee_edit.html` | `EmployeeUpdateView` |
| `/departments/` | `departments.html` | `DepartmentListView` |
| `/leaves/` | `leaves.html` | `LeaveListView` |
| `/uploads/` | `uploads.html` | `UploadStatusView` |
| `/audit/` | `audit.html` | `AuditLogTemplateView` |

### 5.2 Component hierarchy

There is **no component framework**. All templates extend `base.html`
(sidebar + topbar shell). Each page injects its own `<style>` block
(inlining 30-200 lines of CSS per page) and writes raw `fetch()` calls in a
trailing `<script>` block. No re-use, no partials in `templates/partials/`.

### 5.3 State flow

- Server-rendered pages: state lives entirely in Django context dicts. Each
  page roundtrip is a full server reload.
- Quick forms (leave submit, employee create, bulk upload, leave approve)
  POST JSON to the API; on success, `location.reload()` is the universal
  refresh strategy. No optimistic UI, no client-side state library.
- The CSRF token is read from cookie via `getCookie('csrftoken')` and sent
  in `X-CSRFToken`. This is consistent across all JS calls.

### 5.4 Loading / error / empty states

| State | Where handled |
|---|---|
| Loading | A few "Submitting…" / "Uploading…" text mutations. **No spinners** anywhere except the original login button. No skeletons. |
| Error | Status text mutated; some forms `alert()`. JSON errors are dumped via `JSON.stringify(result)` — **raw, ugly, leaks internal structure**. |
| Empty | Each list template has a nice "No employees / no leaves / …" placeholder. ✅ |

### 5.5 Reusable components

None at the JS layer. The only reuse is CSS class names (`.btn`, `.badge`,
`.card`, …) inherited from `base.html`. The `static/js/app.js` file *contains*
reusable classes (`AuthManager`, `APIClient`, `Notification`, `Modal`,
`DataTable`, `FormHandler`) but is **not loaded** by any template — pure dead
code (~640 lines).

### 5.6 Frontend defects (recap)

- **`employee_detail.html:71`** — Django template `{% if %}` cannot contain
  parentheses. Logs show repeated `TemplateSyntaxError: Could not parse the
  remainder: '(user.is_manager'` → **HTTP 500** when viewing an employee.
- **`employee_detail.html:165-167`** — uses `employee.skills.split:";,"` and
  `skill|strip` — neither is valid Django template syntax. The "About &
  Skills" tab silently renders empty.
- **`employee_edit.html:57`** — `{{ employee.phone_number }}` reads from
  `EmployeeProfile` which has no `phone_number` field; the value is on
  `employee.user.phone_number`. The phone input always renders blank for
  existing data (but POSTing it still works because the view sets it on
  `employee.user`).

## 6. Database Analysis

### 6.1 Models (`employees/models.py`)

| Model | PK | Key relations | Notes |
|---|---|---|---|
| `CustomUser` (AbstractUser) | UUID | — | `email` unique; `is_hr_admin`, `is_manager` flags |
| `Department` | UUID | `head → CustomUser (OneToOne, SET_NULL)` | `name` unique, `budget` Decimal |
| `EmployeeProfile` | UUID | `user → CustomUser (OneToOne, CASCADE)`, `department → Department (SET_NULL)`, `manager → CustomUser (SET_NULL)` | `employee_id` unique, `date_of_joining=auto_now_add` |
| `LeaveRequest` | UUID | `employee → EmployeeProfile (CASCADE)`, `approved_by → CustomUser (SET_NULL)` | Has unused `version` field for optimistic locking — actual concurrency uses `select_for_update` instead |
| `AuditLog` | UUID | `user → CustomUser (SET_NULL)` | `changes = JSONField`, content_type stored as plain string (not Django's ContentType FK) |
| `BulkUploadJob` | UUID | `uploaded_by → CustomUser (SET_NULL)` | `error_log = JSONField` (list), `progress_percentage` property |

### 6.2 Indexes

All FK fields plus heavily-queried fields are indexed:
- `CustomUser`: email, is_hr_admin, is_manager
- `Department`: name, is_active
- `EmployeeProfile`: employee_id, employment_status, department, manager
- `LeaveRequest`: (employee, status), (start_date, end_date), status
- `AuditLog`: timestamp, user, object_id, action
- `BulkUploadJob`: status, uploaded_by

### 6.3 Constraint quality

- ✅ `EmployeeProfile.salary` and `bonus` have `MinValueValidator(0)`.
- ✅ `Department.budget` has `MinValueValidator(0)`.
- ❌ No DB-level CHECK that `LeaveRequest.start_date <= end_date`.
- ❌ No DB-level CHECK / unique constraint on `(employee, start_date, end_date)`
   to prevent overlapping leaves (matches the missing serializer validation).
- ❌ `EmployeeProfile.date_of_joining = auto_now_add=True` — locks the joining
   date to row creation time, so CSV-imported employees all share their
   import date as joining date. **Cannot be overridden by HR.**
- ❌ The default migration is named `0001_initial.py` — and there are no
   subsequent migrations even though the codebase clearly edited the model
   (e.g. the `phone_number` field, `is_hr_admin`/`is_manager`, etc., are
   already in `0001_initial`, but if these are evolved, no migration will be
   produced). Always run `python manage.py makemigrations` after model
   changes.

### 6.4 Bugs / smells in models

- `EmployeeProfile.create_employee_profile` defined as a `@receiver` *inside*
  the class body. Works, but creates duplicate profiles when the API path
  also creates one — see §10 Bug #1.
- `BulkUploadJob.progress_percentage` is declared **twice** — the second
  occurrence shadows the first (functionally identical; harmless but smelly).
- `LeaveRequest.version` is declared for "optimistic locking" but never
  read/incremented anywhere. Dead column.
- `AuditLog.content_type` is a plain string; Django provides
  `django.contrib.contenttypes` for the exact same job. Choosing a
  hand-rolled string makes referential integrity weaker.

### 6.5 Query quality

`select_related()` is used consistently on list endpoints:
- `EmployeeProfileViewSet.get_queryset` → `select_related('user','department','manager')`
- `LeaveRequestViewSet.get_queryset` → `select_related('employee__user','employee__department','approved_by')`
- `AuditLogViewSet.get_queryset` → `select_related('user')`

`annotate(Count(..., filter=Q(...)))` is used in `DepartmentViewSet`,
`DashboardView`, and `WorkforceAnalyticsView` for headcount and aggregates.

No `prefetch_related()` is used anywhere — acceptable because there are no
reverse-many queries in list paths.

`AuditLog.objects.bulk_create` is used in `process_csv_file` — good.
`CustomUser.objects.bulk_create` plus `EmployeeProfile.objects.bulk_create` —
good for throughput, but **skips signals** (which is intended here).

## 8. Bugs & Risks

Severity legend: **Critical** = blocks core feature or risks data loss /
security; **High** = correctness issue users will hit; **Medium** = UX or
maintainability; **Low** = style.

### 8.1 Critical

1. **Duplicate `EmployeeProfile` creation on API create** —
   `models.py` line 153-161 has `@receiver(post_save, sender=CustomUser)`
   that auto-creates a profile for any non-superuser. Then
   `EmployeeProfileCreateSerializer.create()` tries to create *another*
   profile for the same user → `IntegrityError`. The "Add Employee" form
   **always fails**. Same root cause breaks `EmployeeProfileFactory` and
   thus most of the test suite.

2. **`employee_detail.html` HTTP 500 on viewing any employee** — the
   `{% if user.is_hr_admin or user == employee.user or (user.is_manager and
   employee.manager == user) %}` uses parentheses, which Django's
   `smart_if` parser rejects. Confirmed by the production
   `logs/django.log`. Affects every employee detail page view.

3. **Leave-approval bypass via PATCH** — `LeaveRequestSerializer` does not
   declare `status` as read-only outside the `approve_leave/reject_leave`
   actions. Any authenticated user can `PATCH
   /api/v1/leave-requests/{their_own_leave_id}/ {"status": "approved"}`
   and self-approve. `select_for_update` only guards the action endpoint;
   the default `update()` path goes through the regular serializer.

4. **`POST /api/v1/auth/register/` is publicly open** — `AllowAny`, no rate
   limiting, and combined with Bug #1 every registration will fail with a
   500 anyway. If Bug #1 is fixed, an attacker can create an unlimited
   number of `CustomUser`s and matching `EmployeeProfile`s with salary=0,
   filling the DB.

5. **`.env` committed with a real DB password** (`DB_PASSWORD=selva`). The
   `.gitignore` does list `.env`, but a copy currently sits in the
   working tree.

### 8.2 High

6. **Duplicate audit log rows on every update / approve** — `views.py`
   calls `log_action()` explicitly **and** `signals.py` writes another row
   via post_save. Audit volume is doubled; the two rows attribute the
   action to different users (request user vs. system).

7. **`DepartmentViewSet.perform_destroy` is not overridden** — deleting a
   department leaves **no audit trail**.

8. **`DELETE /api/v1/leave-requests/{id}/`** has no override and no
   permission narrower than `IsAuthenticated`. Any authenticated user can
   delete leaves in their visible queryset, including approved ones, with
   **no audit log**.

9. **`LeaveRequest` overlap is not validated** — pure `# TODO` in
   `serializers.py:208`. Users can submit overlapping leaves.

10. **`WorkforceAnalyticsView` "approved_this_month"** uses
    `created_at__month=...` only, not year-bounded. Stats reflect all years.

11. **`CanApproveLeaves.has_object_permission`** check
    `hasattr(request.user, 'subordinates')` is always True (it's a reverse
    relation). No real subordinate verification.

12. **`make_random_password()`** in `csv_processor.py` is deprecated in
    Django 5 (removed). Will break on Django 5 upgrade.

13. **`HealthCheckView`** doesn't ping the DB or any dependency — false
    "healthy" responses if the DB is down.

14. **`employee.skills.split:";,"` and `|strip`** in `employee_detail.html`
    are invalid template syntax — skills tab silently empty.

15. **CSV is fully loaded into memory** (`csv_file.read().decode('utf-8')`)
    despite the docstring claim of streaming. For 10k+ row files this is
    fine; for the "100k" scale a resume might claim, it is not.

16. **Two `progress_percentage` properties** on `BulkUploadJob` — harmless
    but indicates the file wasn't reviewed before merging.

17. **`EmployeeProfileSerializer.update()`** writes the `user` sub-object
    *first* then re-saves the profile — no atomic transaction, so a crash
    between the two `save()` calls leaves the user updated and profile
    unchanged.

18. **`LeaveRequestSerializer.validate()` has side effects** (`get_or_create`
    of an `EmployeeProfile`). Validators must be pure; this `EmployeeProfile`
    is *also* created in `perform_create`. The serializer fires the signal
    *before* the view's permission check completes — non-deterministic.

19. **JWT login endpoint is unused** — the resume claim about JWT
    authentication is technically present in code but **not exercised by
    any UI flow**.

### 8.3 Medium

20. **Template duplicates analytics ORM** — `DashboardView`'s queries

21. **`static/js/app.js`** is dead code (640 lines).

22. **`templates/index.html`** has 613 commented-out lines (an earlier

23. **`static/css/temp.css`** duplicates `styles.css` byte-for-byte.

24. **`csv_data/lic.pdf`** is unrelated and committed to the tree.

25. **No CI configuration** — no `.github/workflows`, `.gitlab-ci.yml`,

26. **`filters.py::EmployeeFilter`, `LeaveFilter`, `AuditLogFilter`** are

27. **`CORS_ALLOWED_ORIGINS`** is localhost-only — any real deployment

28. **No rate limiting** anywhere. Login endpoint is brute-forceable.

29. **`pytest.ini --cov-fail-under=70`** but the suite cannot complete

30. **Sample CSV download link is broken** (`/csv_data/sample_bulk_10.csv`

40. **`docs/` is empty**, `templates/partials/` is empty — leftover folders.

## 9. Architecture Recommendations

### 9.1 Immediate fixes (in order of impact)

1. **Move the `CustomUser.post_save` profile-creation receiver** out of
   `models.py` into `signals.py` *and* make it idempotent:
   ```python
   @receiver(post_save, sender=CustomUser)
   def ensure_employee_profile(sender, instance, created, **kwargs):
       if not created or instance.is_superuser:
           return
       EmployeeProfile.objects.get_or_create(
           user=instance,
           defaults={
               'employee_id': generate_employee_id(instance),
               'salary': 0,
           },
       )
   ```
   Then in `EmployeeProfileCreateSerializer.create()` use the same
   `get_or_create` pattern and update the profile in place.

2. **Fix `employee_detail.html` line 71** — remove parentheses, rely on
   `and`'s precedence:
   ```
   {% if user.is_hr_admin or user == employee.user or user.is_manager and employee.manager == user %}
   ```

3. **Lock down `LeaveRequestSerializer`** — add `status`, `approved_by`,
   `approved_by_name`, `approval_date`, `approval_notes` to
   `read_only_fields`; force all state transitions through the action
   endpoints. Also add an `http_method_names = ['get','post','head','options']`
   on `LeaveRequestViewSet` so direct PATCH/DELETE are removed.

4. **Implement leave overlap validation** in the serializer:
   ```python
   if LeaveRequest.objects.filter(
       employee=employee,
       status__in=['pending','approved'],
       start_date__lte=attrs['end_date'],
       end_date__gte=attrs['start_date'],
   ).exists():
       raise serializers.ValidationError({'dates': 'Overlaps an existing leave request'})
   ```
   Add a DB-level CHECK or unique partial index too.

5. **Remove `signals.py::log_employee_profile_changes` and
   `log_leave_request_changes`** OR remove the explicit `log_action()`
   calls in `views.py`. Pick *one* audit-writing strategy. Keeping both
   doubles every entry.

6. **Restrict `UserRegistrationView`** — change permission to
   `IsAuthenticated, IsHRAdmin` (or remove it; the HR flow already creates
   users via `EmployeeProfileCreateSerializer`).

7. **Fix `employee_detail.html` "About & Skills" tab** — replace
   `{% for skill in employee.skills.split:";," %}` with proper handling.
   Use a custom template filter or split on the server in
   `EmployeeDetailView.get_context_data`:
   ```python
   import re
   ctx['skills_list'] = [s.strip() for s in re.split(r'[;,]', employee.skills) if s.strip()]
   ```

8. **Fix `WorkforceAnalyticsView` month filter**: change
   `created_at__month=...` to `created_at__year=now.year,
   created_at__month=now.month`.

9. **Wire `filters.py` FilterSets**: replace each ViewSet's
   `filterset_fields` list with `filterset_class = EmployeeFilter` (etc.).

10. **Add `perform_destroy` to `DepartmentViewSet`** that writes an audit
    log before delete.

11. **Restrict / delete `LeaveRequest` deletion**: override
    `perform_destroy` or set `http_method_names` to forbid DELETE.

12. **Replace `make_random_password()`** with `secrets.token_urlsafe(16)`
    for forward-compatibility.

13. **Real health check**: ping `connection.cursor().execute('SELECT 1')`
    and return `503` on failure.

14. **Delete `static/js/app.js`, `static/css/temp.css`, the dead 613
    lines in `index.html`, `csv_data/lic.pdf`, `templates/partials/`,
    and `docs/`** — or fill them in. Don't ship dead folders.

15. **Strip `.env` of the real password** — replace with
    `DB_PASSWORD=changeme` and commit `.env.example` only.

### 9.2 Refactoring opportunities

- **Split the `employees` app** into `users`, `directory`, `leaves`,
  `audit`, `bulk_imports`. The current god-app makes circular imports
  inevitable as features grow.
- **Move CSV processing to Celery** — the Celery settings are already
  stubbed in `settings.py`. The `process_csv_file()` function is already
  designed to be wrapped in `@shared_task`.
- **Introduce a `services/` layer** between views and models so
  permissions, audit writes, and signals are not all racing each other.
- **Adopt drf-spectacular's** `@extend_schema` decorators per action —
  the auto-generated docs at `/api/v1/docs/` currently show un-decorated
  endpoints with generic descriptions.
- **Replace the `signals.py` thread-local snapshot** with `pre_save`
  capture into the instance itself (`instance._previous = ...`) — same
  semantics, no thread-local state.
- **Adopt `django.contrib.contenttypes.GenericForeignKey`** for
  `AuditLog` so `object_id` is properly type-aware.
- **Replace inline CSS with a single static stylesheet** (you already have
  `static/css/styles.css` — use it).
- **Add a custom template filter** `|split:";"` and `|strip` so existing
  templates work.

### 9.3 Reusable services / utilities to introduce

- `services/leave_service.py` — encapsulates approval transitions,
  overlap check, audit write.
- `services/employee_service.py` — creates user+profile transactionally,
  bypassing the post_save signal.
- `services/audit_service.py` — single source of truth for audit writes
  (delete the duplicate in `signals.py`).
- `utils/csrf.py` — shared CSRF helper for the JS layer (currently
  duplicated five times across templates).

### 9.4 DTO / schema improvements

- Replace `serializers.SerializerMethodField` for `employee_count` with a
  serializer that requires the annotated field — fail loud if the
  queryset wasn't annotated.
- Define explicit `Request` and `Response` serializers for actions
  (`approve_leave` currently swallows `approval_notes` from `request.data`
  without serializer validation).
- Add `BulkUploadJobCreateSerializer` separate from
  `BulkUploadJobSerializer` — current serializer accepts arbitrary write
  fields (mitigated by `read_only_fields`, but DRY-wise should be split).

### 9.5 State management & async

- For the CSV upload status endpoint to be useful, the UI should *poll*
  it instead of reloading the page every 8 seconds.
- For analytics, add Django cache (`cache.get_or_set`) — the dashboard
  re-runs 5-6 aggregate queries per request, which is fine for 50
  employees but will be slow for 5000.
